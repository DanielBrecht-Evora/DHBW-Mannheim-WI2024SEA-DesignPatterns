Visitor Pattern

Design Pattern für erweiterbare Operationen
Was ist das Visitor Pattern?
Neue Operationen auf bestehende Klassen anwenden, ohne diese zu verändern.

Problem vs. Lösung

❌ Ohne Visitor
  • Jede Operation in jede Klasse einbauen
  • Klassen werden immer größer
  • Änderungen an allen Klassen nötig
class Circle { calculateArea(); calculatePerimeter(); printColor(); exportAsSVG(); // immer mehr... }
✅ Mit Visitor
  • Operationen als separate Klassen
  • Shapes bleiben klein und stabil
  • Neue Operation = neue Visitor-Klasse
class Circle { accept(visitor); } class AreaCalculator { visit(Circle c); visit(Square s); }

Wie funktioniert es?

1. Circle circle = new Circle(5);
2. AreaCalculator calc = new AreaCalculator();
↓
3. circle.accept(calc); // Circle ruft calc.visit(this)
↓
4. calc.visit(Circle) wird ausgeführt

Struktur

Shape (Interface)
├─ Circle
├─ Square
└─ Triangle

ShapeVisitor (Interface)
├─ AreaCalculator
├─ PerimeterCalculator
└─ ColorPrinter

Pros & Cons

✅ Vorteile
  • Neue Operationen ohne Klassenänderung
  • Saubere Trennung: Daten ↔ Operationen
  • Verwandte Operationen gruppiert
❌ Nachteile
  • Neue Typen = alle Visitors anpassen
  • Komplexer als einfache Methoden
  • Benötigt accept()-Methode

Wann verwenden?

  • ✅ Viele verschiedene Operationen auf denselben Objekten
  • ✅ Objektstruktur ist stabil (wenige neue Typen)
  • ✅ Originalklassen nicht änderbar
  • ❌ Häufig neue Objekttypen → dann nicht geeignet
