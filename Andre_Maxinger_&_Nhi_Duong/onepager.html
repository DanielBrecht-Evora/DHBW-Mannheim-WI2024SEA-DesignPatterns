Abstract Factory

Factory of Factories

Erzeugungsmuster (Creational Pattern)

Beschreibung

Das Abstract Factory Muster bietet eine Schnittstelle zum Erstellen von Familien verwandter oder abhängiger Produkte, ohne deren konkrete Klassen anzugeben. Es fungiert als "Factory of Factories", indem eine Super-Fabrik weitere Fabriken erstellt, die wiederum bestimmte Produkte erzeugen.

Komponenten

Abstrakte Fabrik Schnittstelle zur Erzeugung abstrakter Produkte einer Produktfamilie
Konkrete Fabrik Implementiert die Regeln der abstrakten Fabrik und erzeugt konkrete Produkte einer Produktfamilie
Abstrakte Produkte Schnittstellen, die alle konkreten Produkte implementieren müssen
Konkrete Produkte Implementieren abstrakte Produkte und gehören zu einer bestimmten Produktfamilie
Client Nutzt die abstrakte Fabrik zum Erstellen von Produktfamilien, ohne deren konkrete Typen anzugeben, und interagiert mit Produkten durch die abstrakten Schnittstellen

Vorteile

  • Das Muster stellt sicher, dass die Anwendung nur Produkte einer Produktfamilie nutzt
  • Lose Kopplung zwischen konkreten Produkten und Client-Code
  • Neue Produktfamilien können einfach hinzugefügt werden
  • Produkterstellung findet an einem zentralen Ort statt

Nachteile

  • Der Code kann komplizierter werden, da viele neue Klassen und Schnittstellen erstellt werden müssen
  • Die Erzeugung neuer Produkttypen benötigt Änderungen an der abstrakten Fabrik und allen konkreten Fabriken

Einsatzgebiete

  • Anwendungen, die mehrere Cloud-Plattformen (z.B. AWS, Azure, Google Cloud) unterstützen müssen
  • Anwendungen, die mehrere Datenbanksysteme (z.B. SQL, NoSQL) unterstützen müssen
  • Plattformübergreifende UI-Toolkits

Warum existiert das Pattern?

Systeme benötigen oft Objektgruppen, die zusammenarbeiten und konsistent bleiben müssen. Ohne Abstract Factory müsste der Client-Code konkrete Klassen direkt instanziieren, was das Wechseln zwischen Produktfamilien erschwert. Die Erstellungslogik wäre über den gesamten Code verteilt, was zu Duplikation führt und das Risiko birgt, inkompatible Produkte verschiedener Familien zu mischen.

Code-Beispiel

1. Abstrakte Fabrik

CatFactory.java
public interface CatFactory { KnightCat createKnightCat(); NinjaCat createNinjaCat(); PirateCat createPirateCat(); }

2. Konkrete Fabrik

MaineCoonCatFactory.java
public class MaineCoonCatFactory implements CatFactory { public KnightCat createKnightCat() { return new MaineCoonKnightCat(); } public NinjaCat createNinjaCat() { return new MaineCoonNinjaCat(); } public PirateCat createPirateCat() { return new MaineCoonPirateCat(); } }

3. Abstraktes Produkt

NinjaCat.java
public interface NinjaCat { void sneak(); }

4. Konkretes Produkt

MaineCoonNinjaCat.java
public class MaineCoonNinjaCat implements NinjaCat { public void sneak() { System.out.println("Maine Coon Ninja sneaks - then knocks over a vase for dramatic effect 🥷"); } }

5. Client-Code

Main.java
public static void main(String[] args) { CatFactory maineCoonFactory = CatFactoryHelper.getFactory(CatFactoryHelper.Breed.MAINE_COON); NinjaCat maineCoonNinjaCat = maineCoonFactory.createNinjaCat(); maineCoonNinjaCat.sneak(); }
